## プロジェクト概要

**リポジトリ**：yamasinnya/ushireclaim
**公開URL**：https://yamasinnya.github.io/ushireclaim/

大分県で黒毛和牛の繁殖農家を営む「やまうし」が制作している牧場育成×探索ゲーム。
実際の繁殖農家の仕事をベースにしつつ、魔石・魔力などのファンタジー要素を加えた作品。

詳細な設計・仕様は以下のドキュメントを参照：
→ やまやま牧場_設計まとめ.md（同リポジトリ内）

---

## 技術スタック

- **言語**：HTML + Vanilla JavaScript
- **探索パート**：Phaser.js（後述のバグ注意）またはピュアJS
- **データ保存**：localStorage
- **公開**：GitHub Pages（静的ファイルのみ、サーバー・DBなし）
- **バックアップ**：ふっかつのじゅもん（Base64エンコード）
- **スタイル**：CSS（フレームワークなし）

やましさんのメインスタックはPHP/Slim4/MySQL/jQueryだが、
このゲームはフロントエンドのみで完結させる方針。
PHPは将来的にセーブデータのサーバー管理が必要になった時点で導入検討。

## 実装済みの主要機能

### 探索パート（神経衰弱）
- 80枚・3種類（コモン/アンコモン/レア）のカード
- 同じ種類でペア成立（どの2枚でもOK）
- 魔力（=ペア試行回数）の管理
- コイン即時獲得（めくった瞬間にチャリン）
- 翌日は盤面の半分をシャッフル、全カードを裏に戻す
- localStorageでフィールド状態を保存

### スキル（探索パート）
| スキル | 効果 |
|---|---|
| 🌀 回転斬り | 1日1回、ランダム9マスを2秒間表示 |
| 👦 牧童の目 | ミスマッチ時の表示時間延長（1.5秒） |
| 🐾 刈り跡の記憶 | 前日マッチしたアンコモン以上が翌日自動収穫 |
| 🔮 第六感 | 毎朝アンコモン3か所がうっすら光る |

### ホーム画面（試作段階）
- 背景画像の上にドット絵スプライトをレイヤーで重ねる構成
- スプライトシート：sprites.png（同リポジトリ内）
- 牛のブチ模様：乱数シードから手続き生成（同一シード→同一模様）

## 実装上の注意（バグ・地雷記録）

### Phaser.js カメラ追従
- `camera.scrollX/Y` の手動計算はズレが生じてキャラが画面外に描画される
- **必ず `camera.startFollow(container)` を使うこと**
- プレイヤーは `this.add.container()` にまとめてから startFollow に渡す

### Phaser.js ダイアログ・UI表示
- `setScrollFactor(0)` はズーム倍率の影響を受けるため、UI要素が画面外に描画される
- **ダイアログ・ステータス表示はPhaserのTextではなくHTMLのdivオーバーレイで実装すること**

### Phaser.js Graphicsオブジェクト
- `g.x = value` で位置を動かすとカリングが誤作動して非表示になることがある
- キャラクター描画は `this.add.rectangle()` などのShapeオブジェクトを使うこと

### localStorageのデータ設計
- セーブキーは必ずバージョン付きで命名する（例：hairal_v3_lv2）
- 牛の個体データには乱数シード（4桁整数）を含める（ブチ模様の再現のため）
- ふっかつのじゅもんはBase64エンコードで実装する

### ファイルの直接開き方（content://）
- Androidでダウンロードしたhtmlをファイルマネージャから開くと content:// URLになる
- Phaser.jsは動作するが一部機能に制限が出る場合がある
- GitHub Pages経由でのアクセスを推奨

## itch.io公開を見据えた実装ルール（2026年7月追記）

将来的にitch.ioでの公開を視野に入れ、以下は**今のうちに**徹底すること。
迷ったらこの節を読み返してから実装する。

### 文字列は直書きしない
画面に表示する日本語は直接コードに書かず、辞書経由で取得する。

```js
t("feed_success")
```

- 中身は `ja.json` のみで良い（英語版は後回し）
- 構造だけ最初から用意しておく

### セーブデータには必ずversionを持たせる

```js
{ version: 1, ... }
```

- 新しいセーブ機能を作る時は必ずこの形式にする
- 将来のセーブデータ変換（マイグレーション）に備える

### 内部名称（変数・プロパティ）は英語で統一

```
health / quality / pregnant / magic
```

- 画面表示のみ日本語、コード内部は英語
- 既存コードで日本語名になっている箇所があれば、触るタイミングで英語に寄せる（無理に一括変更はしない）

### スマホ縦画面・タップ操作が前提

- 縦画面固定で設計する
- タップ操作のみを想定し、マウスオーバー等は考慮しない
- ボタンサイズはタップしやすい大きさを維持する
- Phaser.js実装時は特にこのルールを意識すること（後から直すのが一番痛い）

### イベント・スキルのJSON化はまだしない

- イベント（病気・発情・難産・エピソード等）はまだ種類が固まっていないため、**現時点ではif文でベタ書きでよい**
- 汎用的なJSON駆動エンジンを先に作らない。データ形式が決まっていないものに骨組みを合わせるのは手戻りのリスクが高い
- 目安：似たパターンのイベントが3〜4個実装されて「これはパターン化できる」と気づいた段階で、初めてJSON化を検討する
- スキル（回転斬り等）は実装済みで動いているため、壊れていないものは今は触らない

### 優先度が低いもの（現時点では対応不要）
- BGM実装（効果音を優先、必要ならホーム画面のみ短いループBGMを検討）
- 英語辞書（en.json）の中身（日本語版完成後）
- MOD対応
